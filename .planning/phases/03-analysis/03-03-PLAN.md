Target shape (from 03-RESEARCH.md Pattern 4):
```typescript
import { Router } from 'express';
import { z } from 'zod';
import { uploadMiddleware } from '../middleware/upload.middleware.js';
import { parsePdf } from '../services/pdf.service.js';
import { JdTooLongError } from '../middleware/error.middleware.js';
import { analyzeResume } from '../services/analysis.service.js';

export const analyzeRouter = Router();

const AnalyzeRequestSchema = z.object({
  jobDescription: z.string().trim().min(1).max(5000),
});

analyzeRouter.post('/analyze', ...uploadMiddleware, async (req, res) => {
  // Validate JD BEFORE parsePdf — fail fast on cheap check
  const parsed = AnalyzeRequestSchema.safeParse(req.body);
  if (!parsed.success) {
    throw new JdTooLongError('Job description must be between 1 and 5,000 characters.');
  }

  const resume = await parsePdf(req.file!.buffer);
  const result = analyzeResume(resume, parsed.data.jobDescription);
  res.json(result);
});
```

Integration tests that must now pass (written in Plan 03-01):
- absent jobDescription → 400 jd_too_long
- jobDescription 5001 chars → 400 jd_too_long
- jobDescription exactly 5000 chars → 200
- valid request → { score: number, gaps: string[], rewrites: [], resumeStructure: object }

Note on error code: both missing and too-long jobDescription throw JdTooLongError (code: 'jd_too_long').
The Zod .min(1) handles missing/empty; .max(5000) handles too-long. One error class covers both.
This is intentional per the research — keep the error surface simple.
